## 上下文与记忆管理

要在会话间共享记忆，最好的方式是创建一个技能或命令来总结和检查进度，然后保存到 `.claude` 文件夹中的 `.tmp` 文件，并在会话结束前持续追加。第二天可以将其作为上下文继续工作，为每个会话创建新文件以避免旧上下文污染新工作。最终你会有一个包含大量会话日志的文件夹——只需备份到有意义的地方或修剪不需要的会话对话。

Claude 会创建一个总结当前状态的文件。审查它，如果需要就要求修改，然后重新开始。对于新对话，只需提供文件路径。当你达到上下文限制并需要继续复杂工作时特别有用。这些文件应包含：哪些方法有效（有证据可验证），尝试过但无效的方法，未尝试过的方法以及剩余工作。

**战略性清理上下文：**

一旦你的计划确定且上下文已清理（Claude Code 中计划模式的默认选项），你就可以按照计划工作。当你积累了大量与执行不再相关的探索上下文时，这很有用。对于战略性压缩，禁用自动压缩。在逻辑间隔处手动压缩，或创建一个技能来为你执行或根据定义的标准建议压缩。

**战略性压缩技能（直接链接）：** [strategic-compact](https://github.com/affaan-m/everything-claude-code/tree/main/skills/strategic-compact)

将其挂接到 Edit/Write 操作的 PreToolUse 钩子——当你积累了足够的上下文时，它会提醒你压缩可能会有帮助。

**高级：动态系统提示注入**

我学到并正在试运行的一个模式是：不要仅仅把所有内容放在 CLAUDE.md（用户范围）或 `.claude/rules/`（项目范围）中（这些每次会话都会加载），而是使用 CLI 标志动态注入上下文。

```bash
claude --system-prompt "$(cat memory.md)"
```

这让你可以更精确地控制何时加载什么上下文。你可以根据当前工作内容为每个会话注入不同的上下文。

**为什么这比 @ 文件引用更重要：**

当你使用 `@memory.md` 或将内容放在 `.claude/rules/` 中时，Claude 在对话中通过 Read 工具读取它——它作为工具输出进入。当你使用 `--system-prompt` 时，内容在对话开始前被注入到实际的系统提示中。

区别在于指令层次结构。系统提示内容的优先级高于用户消息，用户消息的优先级高于工具结果。对于大多数日常工作来说，这种差异微乎其微。但对于严格的行为规则、项目特定约束或你绝对需要 Claude 优先处理的上下文——系统提示注入确保其获得适当的权重。

**实际设置：**

一种有效的方式是利用 `.claude/rules/` 作为基线项目规则，然后为特定场景的上下文创建 CLI 别名：

```bash
# 日常开发
alias claude-dev='claude --system-prompt "$(cat ~/.claude/contexts/dev.md)"'

# PR 审查模式
alias claude-review='claude --system-prompt "$(cat ~/.claude/contexts/review.md)"'

# 研究/探索模式
alias claude-research='claude --system-prompt "$(cat ~/.claude/contexts/research.md)"'
```

- `dev.md` 聚焦于实现
- `review.md` 聚焦于代码质量/安全
- `research.md` 聚焦于行动前的探索

对于大多数场景，使用 `.claude/rules/context1.md` 和直接将 `context1.md` 附加到系统提示之间的差异微乎其微。CLI 方式更快（无需工具调用）、更可靠（系统级权限）、token 效率略高。但这是一个小优化，对很多人来说开销可能不值得。

**高级：记忆持久化钩子**

有一些大多数人不知道或知道但没有充分利用的钩子可以帮助记忆：

- **PreCompact 钩子：** 上下文压缩前，将重要状态保存到文件
- **SessionComplete 钩子：** 会话结束时，将学到的知识持久化到文件
- **SessionStart 钩子：** 新会话时，自动加载之前的上下文

**记忆持久化钩子（直接链接）：** [memory-persistence](https://github.com/affaan-m/everything-claude-code/tree/main/hooks/memory-persistence/)

```json
{
  "hooks": {
    "PreCompact": [{
      "matcher": "*",
      "hooks": [{
        "type": "command",
        "command": "~/.claude/hooks/memory-persistence/pre-compact.sh"
      }]
    }],
    "SessionStart": [{
      "matcher": "*",
      "hooks": [{
        "type": "command",
        "command": "~/.claude/hooks/memory-persistence/session-start.sh"
      }]
    }],
    "Stop": [{
      "matcher": "*",
      "hooks": [{
        "type": "command",
        "command": "~/.claude/hooks/memory-persistence/session-end.sh"
      }]
    }]
  }
}
```

**这些钩子的功能：**

- **pre-compact.sh：** 记录压缩事件，用压缩时间戳更新活跃会话文件
- **session-start.sh：** 检查最近的会话文件（过去7天），通知可用的上下文和学到的技能
- **session-end.sh：** 创建/更新每日会话文件，跟踪开始/结束时间

将这些链接在一起，实现无需手动干预的跨会话连续记忆。这建立在第一篇文章中的钩子类型（PreToolUse、PostToolUse、Stop）之上，但专门针对会话生命周期。

## 持续学习 / 记忆

我们讨论了以更新 codemap 形式的连续记忆更新，但这也适用于其他事情，比如从错误中学习。如果你不得不多次重复同一个提示，而 Claude 遇到了同样的问题或给了你之前听过的回答，这对你适用。

你很可能需要发出第二个提示来"重新引导"和校准 Claude 的指南针。这适用于任何此类场景——这些模式必须追加到技能中。

现在你可以自动完成这个过程，只需告诉 Claude 记住它或将其添加到你的规则中，或者你可以创建一个专门做这件事的技能。

**问题：** 浪费的 token、浪费的上下文、浪费的时间，你的皮质醇飙升，因为你沮丧地对 Claude 大喊不要做某件你在之前的会话中已经告诉它不要做的事。

**解决方案：** 当 Claude Code 发现一些不平凡的东西——调试技术、解决方案、某些项目特定的模式——它会将该知识保存为新技能。下次出现类似问题时，技能会自动加载。

**持续学习技能（直接链接）：** [continuous-learning](https://github.com/affaan-m/everything-claude-code/tree/main/skills/continuous-learning)

```bash
# 克隆到技能文件夹
git clone https://github.com/affaan-m/everything-claude-code.git ~/.claude/skills/everything-claude-code

# 或者只获取持续学习技能
mkdir -p ~/.claude/skills/continuous-learning
curl -sL https://raw.githubusercontent.com/affaan-m/everything-claude-code/main/skills/continuous-learning/evaluate-session.sh > ~/.claude/skills/continuous-learning/evaluate-session.sh
chmod +x ~/.claude/skills/continuous-learning/evaluate-session.sh
```

**为什么我用 Stop 钩子而不是 UserPromptSubmit？** UserPromptSubmit 在你发送的每条消息上运行——开销很大，给每个提示增加延迟，坦率地说对于这个目的来说过度了。Stop 在会话结束时运行一次——轻量级，不会在会话期间拖慢你，并且评估的是完整会话而不是零散片段。

**安装：**

**钩子配置（直接链接）：**

```json
{
  "hooks": {
    "Stop": [{
      "matcher": "*",
      "hooks": [{
        "type": "command",
        "command": "~/.claude/skills/continuous-learning/evaluate-session.sh"
      }]
    }]
  }
}
```

这使用 Stop 钩子在每个提示上运行激活脚本，评估会话中值得提取的知识。该技能也可以通过语义匹配激活，但钩子确保一致的评估。

Stop 钩子在你的会话结束时触发——脚本分析会话中值得提取的模式（错误解决、调试技术、解决方案、项目特定模式等），并将它们保存为 `~/.claude/skills/learned/` 中的可复用技能。

**使用 /learn 手动提取：**

你不必等到会话结束。仓库还包括一个 `/learn` 命令，你可以在会话中当你刚刚解决了不平凡的问题时运行它。它会提示你立即提取模式，起草技能文件，并在保存前请求确认。

**会话日志模式：**

该技能期望 `.tmp` 文件中的会话日志。模式是：`~/.claude/sessions/YYYY-MM-DD-topic.tmp` —— 每个会话一个文件，包含当前状态、已完成项目、阻碍因素、关键决策和下次会话的上下文。示例会话文件在仓库的 [examples/sessions/](https://github.com/affaan-m/everything-claude-code/tree/main/examples/sessions/) 中。

**其他自我改进的记忆模式：**

来自 @RLanceMartin 的一种方法涉及反思会话日志以提炼用户偏好——本质上是建立一个"日记"记录什么有效什么无效。每次会话后，反思代理提取做得好的、失败的、你做了哪些纠正。这些学习更新一个在后续会话中加载的记忆文件。

来自 @alexhillman 的另一种方法让系统每15分钟主动建议改进，而不是等你自己发现模式。代理审查最近的交互，提出记忆更新，你批准或拒绝。随着时间推移，它会从你的批准模式中学习。

## Token 优化

我收到了很多来自价格敏感消费者或经常遇到限制问题的重度用户的问题。在 token 优化方面，你可以做一些技巧。

**主要策略：子代理架构**

主要是优化你使用的工具和子代理架构，设计为将最便宜的足够完成任务的模型委派出去以减少浪费。你有几个选项——你可以尝试试错并随进展调整。一旦你学会了什么是什么，你可以决定哪些可以委派给 Haiku，哪些可以委派给 Sonnet，哪些可以委派给 Opus。

**基准测试方法（更复杂）：**

另一种更复杂的方式是让 Claude 设置一个基准测试，你有一个定义明确的目标和任务以及明确定义的计划的仓库。在每个 git worktree 中，让所有子代理使用一个模型。记录任务完成情况——理想情况下在你的计划和任务中。你必须至少使用每个子代理一次。

完成一轮完整通过并且任务在你的 Claude 计划中被勾选后，停止并审计进度。你可以通过比较差异、创建跨所有 worktree 统一的单元测试、集成测试和 E2E 测试来完成。这将给你一个基于通过用例与失败用例的数值基准。如果所有测试都通过，你需要添加更多测试边界情况或增加测试的复杂度。这可能值得也可能不值得，取决于这对你的实际重要程度。

**模型选择快速参考：**

90% 的编码任务默认使用 Sonnet。当第一次尝试失败、任务跨越5个以上文件、架构决策或安全关键代码时升级到 Opus。当任务是重复性的、指令非常清晰或在多代理设置中作为"工作者"时降级到 Haiku。坦率地说，Sonnet 4.5 目前处于一个奇怪的位置，每百万输入 token 3 美元，每百万输出 token 15 美元，相比 Opus 节省约 66.7%，绝对来说是不错的节省，但相对而言对大多数人来说或多或少微不足道。Haiku 和 Opus 的组合最有意义，因为 Haiku 与 Opus 是 5 倍的成本差异，而与 Sonnet 相比只有 1.67 倍的价格差异。

在你的代理定义中，指定模型：

```yaml
---
name: quick-search
description: Fast file search
tools: Glob, Grep
model: haiku # 便宜且快速
---
```

**工具特定优化：**

想想 Claude 最频繁调用的工具。例如，用 mgrep 替换 grep——在各种任务中，与 Claude 默认使用的传统 grep 或 ripgrep 相比，mgrep 的有效 token 减少平均约一半。

**后台进程：**

在适用的情况下，如果你不需要 Claude 处理整个输出并且不需要实时流式传输，在 Claude 外部运行后台进程。这可以轻松通过 tmux 实现（参见简明指南和 [Tmux 命令参考](https://tmuxcheatsheet.com/)）。获取终端输出，要么总结它，要么只复制你需要的部分。这将节省大量输入 token，这是成本的主要来源——Opus 4.5 每百万 token 5 美元，输出每百万 token 25 美元。

**模块化代码库的好处：**

拥有更模块化的代码库，包含可复用的实用工具、函数、钩子等——主文件保持在几百行而不是几千行——有助于 token 优化成本和第一次就正确完成任务，这两者是相关的。如果你不得不多次提示 Claude，你会消耗大量 token，特别是当它反复读取非常长的文件时。你会注意到它需要做很多工具调用才能完成读取文件。在这个过程中，Claude 可能会丢失一些信息。而且，停止和重新读取会消耗额外的 token。这可以通过拥有更模块化的代码库来避免。

**精简代码库 = 更便宜的 Token：**

这可能显而易见，但你的代码库越精简，你的 token 成本就越低。通过使用技能持续清理代码库来进行重构来识别死代码至关重要。在某些时候，我喜欢浏览整个代码库，寻找对我来说突出或看起来重复的内容，手动拼凑上下文，然后将其与重构技能和死代码技能一起输入 Claude。

**系统提示瘦身（高级）：**

对于真正注重成本的人：Claude Code 的系统提示约占 18k token（200k 上下文的 9%）。通过补丁可以减少到约 10k token，节省约 7,300 token（静态开销的 41%）。如果你想走这条路，参见 YK 的 system-prompt-patches，我个人不做这个。

## 验证循环和评估

评估和调优——根据项目，你需要使用某种形式的可观察性和标准化。

**可观察性方法：**

一种方式是让 tmux 进程挂接到跟踪思考流和每当技能被触发时的输出。另一种方式是有一个 PostToolUse 钩子，记录 Claude 具体执行了什么以及确切的更改和输出是什么。

**基准测试工作流：**

将其与在没有技能的情况下要求同一任务并检查输出差异以基准测试相对性能进行比较：

分叉对话，在其中一个中启动一个新的 worktree 但不使用该技能，最后拉出差异，查看记录了什么。这与持续学习和记忆部分相关联。

**评估模式类型：**

更高级的评估和循环协议在这里进入。分裂点是基于检查点的评估和基于 RL 任务的连续评估。

**基于检查点的评估：**

- 在工作流中设置明确的检查点
- 在每个检查点根据定义的标准验证
- 如果验证失败，Claude 必须修复后才能继续
- 适用于具有明确里程碑的线性工作流

**连续评估：**

- 每 N 分钟或重大更改后运行
- 完整测试套件、构建状态、lint
- 立即报告回归
- 停止并修复后再继续
- 适用于长时间运行的会话

决定因素是你的工作性质。基于检查点适用于具有明确阶段的功能实现。连续评估适用于探索性重构或没有明确里程碑的维护。

我认为在某些干预下，验证方法足以避免大多数技术债务。让 Claude 在完成任务后通过运行技能和 PostToolUse 钩子进行验证有助于此。持续的 codemap 更新也有帮助，因为它记录了更改和 codemap 如何随时间演变，作为仓库本身之外的真实来源。有了严格的规则，Claude 也会避免创建散乱的 .md 文件、重复的类似代码文件以及留下死代码的荒地。

**评分器类型（来自 Anthropic - 直接链接）：[Demystifying evals for AI agents](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents)**

- **基于代码的评分器：** 字符串匹配、二元测试、静态分析、结果验证。快速、便宜、客观，但对有效变化脆弱。
- **基于模型的评分器：** 评分标准评分、自然语言断言、成对比较。灵活且处理细微差别，但非确定性且更昂贵。
- **人工评分器：** SME 审查、众包判断、抽样检查。黄金标准质量，但昂贵且缓慢。

**关键指标：**

- **pass@k：** k 次尝试中至少有一次成功。k 越高 = 成功率越高。
- **pass^k：** 所有 k 次尝试都必须成功。k 越高 = 越难（一致性）。

当你只需要它能工作且任何验证反馈就足够时使用 pass@k。当一致性至关重要且你需要近乎确定性的输出一致性（在结果/质量/风格方面）时使用 pass^k。

**构建评估路线图（来自同一份 Anthropic 指南）：**

1. 尽早开始——来自真实失败的 20-50 个简单任务
2. 将用户报告的失败转化为测试用例
3. 编写明确的任务——两个专家应该得出相同的结论
4. 构建平衡的问题集——测试行为应该和不应该发生的情况
5. 构建健壮的测试工具——每次试验从干净的环境开始
6. 评估代理产生的结果，而不是它采取的路径
7. 阅读多次试验的记录
8. 监控饱和度——100% 通过率意味着需要添加更多测试

## 并行化

在多 Claude 终端设置中分叉对话时，确保分叉和原始对话中的操作范围明确。在代码更改方面力求最小重叠。选择彼此正交的任务以防止干扰的可能性。

**我的首选模式：**

就个人而言，我更喜欢主聊天处理代码更改，而我做的分叉用于关于代码库及其当前状态的问题，或研究外部服务如拉取文档、搜索适用于任务的开源仓库或其他有帮助的一般研究。

**关于任意终端数量：**

Boris @bcherny（创建 Claude Code 的传奇人物）有一些关于并行化的建议，我同意也不同意。他建议过类似在本地运行5个 Claude 实例和在上游运行5个。我建议不要设置这样任意的终端数量。添加终端和添加实例应该出于真正的需求和目的。如果你能用脚本处理那个任务，就用脚本。如果你能在主聊天中让 Claude 在 tmux 中启动一个实例并在单独的终端中流式传输，就那样做。

你的目标真的应该是：用最小可行的并行化完成尽可能多的工作。

对于大多数新手，我甚至建议在你掌握单个实例并在其中管理一切之前远离并行化。我不是在提倡限制自己——我只是说要小心。大多数时候，即使我也只使用大约4个终端。我发现我通常只用2或3个 Claude 实例就能做大多数事情。

**扩展实例时：**

如果你要开始扩展实例并且你有多个 Claude 实例处理彼此重叠的代码，必须使用 git worktree 并为每个实例制定非常明确的计划。此外，为了在恢复会话时不会因为哪个 git worktree 用于什么而困惑或迷失（除了树的名称之外），使用 `/rename <名称>` 来命名你所有的聊天。

**Git Worktree 用于并行实例：**

```bash
# 为并行工作创建 worktree
git worktree add ../project-feature-a feature-a
git worktree add ../project-feature-b feature-b
git worktree add ../project-refactor refactor-branch

# 每个 worktree 有自己的 Claude 实例
cd ../project-feature-a && claude
```

**好处：**

- 实例间无 git 冲突
- 每个都有干净的工作目录
- 易于比较输出
- 可以跨不同方法基准测试同一任务

**级联方法：**

运行多个 Claude Code 实例时，使用"级联"模式组织：

- 在右侧新标签页中打开新任务
- 从左到右扫描，从最旧到最新
- 保持一致的方向流
- 根据需要检查特定任务
- 同时最多关注 3-4 个任务——超过这个数量，心理开销的增长速度会超过生产力

## 基础工作

从头开始时，实际的基础很重要。这应该显而易见，但随着代码库的复杂性和规模增加，技术债务也会增加。管理它极其重要，如果你遵循几条规则，也不那么困难。除了为当前项目有效设置 Claude（参见简明指南）。

**双实例启动模式：**

对于我自己的工作流管理（非必要但有帮助），我喜欢用2个打开的 Claude 实例启动一个空仓库。

**实例1：脚手架代理**

- 铺设脚手架和基础工作
- 创建项目结构
- 设置配置（CLAUDE.md、规则、代理——简明指南中的所有内容）
- 建立约定
- 搭建骨架

**实例2：深度研究代理**

- 连接你所有的服务、网络搜索等
- 创建详细的 PRD
- 创建架构 mermaid 图
- 编译带有实际文档片段的引用

最少需要什么就开始——这比每次都用 Context7 或输入链接让它抓取或使用 Firecrawl MCP 站点更快。当你已经深入某事并且 Claude 明显搞错了语法或使用了过时的函数或端点时，所有这些都有效。

**llms.txt 模式：**

如果可用，你可以在许多文档参考中通过到达文档页面后执行 `/llms.txt` 来找到 llms.txt。这是一个例子：https://www.helius.dev/docs/llms.txt

这给你一个干净的、LLM 优化的文档版本，可以直接输入 Claude。

**哲学：构建可复用模式**

来自 @omarsar0 的一个我完全认同的洞察："早期，我花时间构建可复用的工作流/模式。构建起来很繁琐，但随着模型和代理工具的改进，这产生了疯狂的复合效应。"

**值得投资的：**

- 子代理（简明指南）
- 技能（简明指南）
- 命令（简明指南）
- 规划模式
- MCP 工具（简明指南）
- 上下文工程模式

**为什么会有复合效应（@omarsar0）：** "最棒的部分是所有这些工作流都可以转移到其他代理如 Codex。"一旦构建完成，它们可以跨模型升级工作。投资模式 > 投资特定模型技巧。

## 代理与子代理的最佳实践

在简明指南中，我列出了子代理结构——planner、architect、tdd-guide、code-reviewer 等。在这一部分，我们专注于编排和执行层。

**子代理上下文问题：**

子代理的存在是为了通过返回摘要而不是转储所有内容来节省上下文。但编排器拥有子代理缺乏的语义上下文。子代理只知道字面查询，不知道请求背后的目的/推理。摘要经常遗漏关键细节。

来自 @PerceptualPeak 的类比："你的老板派你去开会并要求总结。你回来给他汇报。十有八九，他会追问。你的摘要不会包含他需要的一切，因为你没有他拥有的隐含上下文。"

**迭代检索模式：**

为了解决这个问题，让编排器：

1. 评估每个子代理返回
2. 在接受之前提出后续问题
3. 子代理回到源头，获取答案，返回
4. 循环直到足够（最多3个循环以防止无限循环）

**传递客观上下文，而不仅仅是查询。** 分派子代理时，包含具体查询和更广泛的目标。这帮助子代理优先考虑在摘要中包含什么。

**模式：具有顺序阶段的编排器**

```markdown
阶段1：研究（使用 Explore 代理）
- 收集上下文
- 识别模式
- 输出：research-summary.md

阶段2：计划（使用 planner 代理）
- 读取 research-summary.md
- 创建实现计划
- 输出：plan.md

阶段3：实现（使用 tdd-guide 代理）
- 读取 plan.md
- 先写测试
- 实现代码
- 输出：代码更改

阶段4：审查（使用 code-reviewer 代理）
- 审查所有更改
- 输出：review-comments.md

阶段5：验证（如需要使用 build-error-resolver）
- 运行测试
- 修复问题
- 输出：完成或循环回去
```

**关键规则：**

1. 每个代理获得一个明确的输入并产生一个明确的输出
2. 输出成为下一阶段的输入
3. 永远不要跳过阶段——每个阶段都增加价值
4. 在代理之间使用 `/clear` 保持上下文新鲜
5. 将中间输出存储在文件中（而不仅仅是记忆中）

**代理抽象层级（来自 @menhguin）：**

**第一层：直接增益（易于使用）**

- **子代理** - 防止上下文腐化和临时专业化的直接增益。多代理一半的有用性但复杂性低得多
- **元提示** - "我花3分钟提示一个20分钟的任务。"直接增益——提高稳定性并检查假设
- **开始时多问用户** - 通常是增益，虽然你需要在计划模式中回答问题

**第二层：高技能门槛（更难用好）**

- **长时间运行的代理** - 需要理解15分钟任务与1.5小时与4小时任务的形态和权衡。需要一些调整，显然是非常长的试错
- **并行多代理** - 方差非常高，仅在高度复杂或良好分段的任务上有用。"如果2个任务需要10分钟而你花了任意数量的时间提示或者更糟的是合并更改，那就是事倍功半"
- **基于角色的多代理** - "模型进化太快，不适合硬编码启发式，除非套利非常高。"难以测试
- **计算机使用代理** - 非常早期的范式，需要驾驭。"你在让模型做一年前它们绝对没有被设计来做的事情"

**要点：** 从第一层模式开始。只有在掌握了基础并有真正需求时才进阶到第二层。

## 技巧和窍门

**一些 MCP 是可替代的，可以释放你的上下文窗口**

方法如下。

对于版本控制（GitHub）、数据库（Supabase）、部署（Vercel、Railway）等 MCP——这些平台大多已经有健壮的 CLI，MCP 本质上只是包装器。MCP 是不错的包装器，但它有成本。

为了让 CLI 功能更像 MCP 而不实际使用 MCP（以及随之而来的减少的上下文窗口），考虑将功能打包到技能和命令中。剥离 MCP 暴露的使事情变得简单的工具，将它们变成命令。

示例：与其始终加载 GitHub MCP，不如创建一个 `/gh-pr` 命令来包装 `gh pr create` 并使用你首选的选项。与其让 Supabase MCP 消耗上下文，不如创建直接使用 Supabase CLI 的技能。功能相同，便利性相似，但你的上下文窗口被释放出来用于实际工作。

这与我收到的其他一些问题相关。自从我发布原始文章以来的几天里，Boris 和 Claude Code 团队在内存管理和优化方面取得了很大进展，主要是 MCP 的懒加载，这样它们不再从一开始就消耗你的窗口。以前我会建议尽可能将 MCP 转换为技能，将功能卸载以通过两种方式之一使用 MCP：在那时启用它（不太理想，因为你需要离开并恢复会话）或使用与 MCP 的 CLI 类比的技能（如果存在）并让技能作为包装器——本质上让它充当伪 MCP。

有了懒加载，上下文窗口问题基本解决了。但 token 使用和成本没有以同样的方式解决。CLI + 技能方法仍然是一种 token 优化方法，其效果可能与使用 MCP 相当或接近。此外，你可以通过 CLI 而不是在上下文中运行 MCP 操作，这显著减少 token 使用，对于数据库查询或部署等重度 MCP 操作特别有用。
